01 — The Problem

The ROI agent missed levers and required extensive rework

Consultants reported spending 1-2 hours prompting the agent before the output was client-ready. The root cause was not about the financial modeling — it was about how levers were identified.

What the agent did
  • Jumped straight from evidence to financial model
  • Picked levers opportunistically from transcript mentions
  • Missed levers that weren't explicitly stated in evidence
  • No problem statement, no hypothesis tree, no MECE structure
  • Crammed 17K tokens of instructions into one agent's context
What a consultant does
  • Defines the problem statement first
  • Builds a MECE hypothesis tree from the problem
  • Derives levers systematically from the tree
  • Then scans data to size each lever
  • Uses benchmarks where client data is missing

03 — Foundation: What is a Value Lever?

Every lever must have four links

If any link is missing, it's not a lever — it's either a wish or a feature description.

Root Driver

Observable problem in the bank's operations, sourced from evidence

→

Operational Change

Specific Backbase capability that changes the process

→

Volume/Rate Impact

Measurable metric that moves, with gap-based derivation

→

Financial Impact

Dollar value calculated from bank's own data

Example (BECU): 26.1% abandonment rate (Root Driver) → Digital Onboarding Lifecycle DOL.1-5 with save-and-resume, eKYC (Operational Change) → 18.1pp gap × 0.40 capture = 7.2pp improvement (Volume/Rate) → recovered apps × funding rate × $350 revenue = annual uplift (Financial Impact)

Model: Opus (strongest reasoning)

Model: Sonnet (efficient for structured work)

Job: Size each lever using gap-based methodology, build 3-scenario model

Reads: Validated levers + benchmarks + client data

Produces: roi_config.json + roi_report.md

Does NOT: Identify levers or build hypothesis trees

05 — Why Two Agents?

Context isolation, enforced sequence, better results

Context Isolation

Agent 1 loads methodology (~700 lines). Agent 2 loads financial data + config schema. Neither wastes context on the other's content. Old agent: 17K tokens. New: 5K + 10K in separate sessions.

Enforced Sequence

The orchestrator physically cannot launch Agent 2 until Agent 1's output file exists. This is a structural guarantee that the problem statement and hypothesis tree are built before any numbers.

Turn Efficiency

Each agent is focused. No wasted turns on wrong-track work. Agent 1 finishes in ~10 min, Agent 2 in ~15 min. Total comparable to the old single agent, but better results.

Consultant Checkpoint

Lever candidates are presented for validation BEFORE any financial modeling. This is where the consultant catches missed levers — before hours of rework, not after.

06 — Foundation Documents

Three knowledge documents power the new pipeline

Definition

value_lever_framework.md

Defines the four-link chain with real examples from BECU (Raghu-validated), NFIS, HNB, and SeABank. Includes validation criteria, confidence levels, and common failure modes.

Top-Down Patterns

hypothesis_tree_decomposition.md

MECE decomposition patterns for 6 problem types (Revenue, Cost, Digital, Efficiency, Lending, AUM) with LOB-specific elaborations across Retail, SME, Commercial, Corporate, Wealth, and Investing.

Bottom-Up Patterns

capability_driven_patterns.md

For exercises starting from a platform component (Grand Central, APA, Managed Hosting, Data Foundations, AI & Agentic). Traces value chains from capability up to business outcomes.

Design decision: These are consultant-facing documents. The agent loads only the relevant one at runtime — not all three. This keeps context lean.

09 — Orchestrator Integration

How the pipeline runs

## Non-interactive mode (full pipeline)

Discovery →
  Block A1: [Journey, Market, Capability, ROI-Hypothesis, Benchmark]  # 5 parallel
  → Auto-approve checkpoints
  Block A2: [ROI-Financial-Model]  # Sequential, reads A1 outputs
  → ROI validation gate
→ Roadmap → Assembly → HTML + Excel
## Standalone mode (/build-roi skill)

Consultant provides: bank name + LOB + problem statement
→ roi-hypothesis-builder → lever_candidates.md
→ Consultant validates levers
→ roi-financial-modeler → roi_config.json + roi_report.md
→ /generate-roi-excel → Excel workbook
Downstream tools unchanged: roi_excel_generator.py and roi_calibrator.py consume the same roi_config.json schema. No changes required.

11 — Known Gaps & Next Steps

What still needs work

Training Data Gaps

SME & Corporate LOBs

No complete ROI models for SME or Corporate banking. Layer 2 patterns are benchmark-based estimates. Need 1+ real engagement per LOB to validate.

Schema Compliance

Financial Modeler Output Format

Agent 2 sometimes produces arrays where the Excel generator expects keyed dicts. Explicit schema examples have been added to the agent prompt. Monitor on next test runs.

Not Yet Covered

Additional Capability Patterns

Digital Engage (standalone), Identity layer, and product edition-specific (Essential/Premium/Signature) patterns not yet written.

Pending

Regression Testing

Run against HNB (Wealth) and NFIS (Investing) evidence registers to validate cross-LOB coverage matches or exceeds existing outputs.
